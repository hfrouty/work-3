## 1. 專案名稱

專案名稱：**PickEat**

PickEat 是一個「今天吃什麼推薦網站」。使用者可以設定搜尋地點、價格區間、距離，並輸入今天想吃什麼，網站會推薦附近適合的餐廳，幫助有選擇困難的人更快決定一餐。

這個專案從一開始的學生專題 MVP 出發，先用示範資料完成核心流程，後來逐步加入後端、Google Places API、OpenAI AI 推薦、Google 地圖圖釘、收藏、歷史紀錄、扭蛋隨機抽等功能，讓它更接近可以實際展示的餐廳推薦產品。

---

## 2. 專案動機：為什麼想做這個專案

### 生活中常遇到的問題

很多人在日常生活中都會遇到「不知道今天要吃什麼」的問題。尤其是學生下課後、讀書到一半、準備和朋友聚餐時，常常會在聊天群組裡問：

- 今天吃什麼？
- 要不要吃便宜一點？
- 附近有什麼可以快速解決？
- 想喝熱湯，但不要太遠。
- 有沒有適合一個人吃的店？

表面上只是選餐廳，但其實背後包含很多條件，例如地點、距離、價格、用餐情境、朋友人數、心情、天氣、時間，以及個人口味。這些條件如果全部自己查，會花很多時間。

### 這個問題為什麼值得解決

吃飯是每天都會發生的需求，因此「快速決定吃什麼」是一個很貼近日常生活的題目。它不只是做一個靜態網頁，而是可以設計成一個互動式工具：使用者輸入條件，系統回傳推薦結果，並提供地圖與導航。

這個題目適合學生專題，原因是：

- 問題很生活化，老師和同學都能理解。
- 功能可以從簡單版本開始做，再慢慢擴充。
- 可以結合前端互動、資料篩選、API、AI、地圖、部署等技術。
- 很適合用 MVP 的方式展示「從想法到產品」的開發過程。

### 目標使用者

PickEat 的主要使用者可以設定為：

- 有選擇困難的學生。
- 下課後想快速找餐廳的人。
- 不想自己慢慢打開 Google Maps 比較店家的人。
- 有價格區間限制的人。
- 想依照距離、地點、需求快速找店的人。
- 和朋友聚餐但不知道要去哪裡的人。

### 為什麼適合做成網站

網站不需要安裝 App，只要打開網址就能使用，很適合課堂展示，也方便同學、老師、組員直接在手機或電腦上測試。餐廳推薦也天然適合做成網頁互動流程：輸入條件、搜尋、推薦、收藏、查看地圖、開始導航。

### 和課堂內容、AI、網頁應用的關聯

這個專案可以連結到多個課堂或資訊應用概念：

- **前端開發**：React 元件拆分、狀態管理、表單互動、RWD。
- **資料結構**：餐廳資料欄位、價格區間、關鍵字、歷史紀錄。
- **API 串接**：Google Places、Geocoding、Google Maps JavaScript API、Google Maps URL。
- **後端概念**：用 Express 保護 API Key，不把金鑰放在前端。
- **AI 應用**：OpenAI 用來解析自然語言需求、協助排序、產生推薦原因。
- **產品思維**：先做 MVP，再依照使用者需求逐步修正。

## 3. 專案目標

### 原本想解決的問題

PickEat 最初想解決的是：使用者每天要吃飯，但常常不知道該選哪間店。網站希望透過地點、距離、價格和需求文字，幫使用者快速縮小選項。

### 使用者進入網站後可以做什麼

目前使用者可以：

- 使用定位取得目前位置。
- 選擇台灣縣市。
- 在 Google 地圖上拖曳圖釘，設定搜尋中心。
- 輸入搜尋所在地，例如「彰師大」、「台北 101」。
- 選擇價格區間或自訂價格上限。
- 選擇距離或自訂距離。
- 輸入今天想吃什麼。
- 使用快速需求選單加入標籤。
- 按下「幫我推薦」取得最多 4 間餐廳推薦。
- 使用「換一批」取得另一組餐廳。
- 使用扭蛋隨機抽模式抽出一間餐廳。
- 收藏餐廳。
- 查看最近 5 次推薦歷史。
- 點擊「查看地圖」或「開始導航」。

### 網站希望提供的價值

PickEat 不是單純列出餐廳，而是希望把「選擇」這件事變簡單。它把使用者原本需要自己考慮的條件，例如距離、價格、營業狀態、評分、需求文字，整理成推薦結果，讓使用者可以更快決定。

### 最初預期完成的 MVP 功能

最初的 MVP 目標是：

- 建立 React + Vite + JavaScript 專案。
- 使用前端假資料，不接資料庫、不接後端、不接正式 API。
- 建立至少 20 筆彰化市風格的虛構餐廳資料。
- 可以依照價格區間、距離與需求文字推薦餐廳。
- 可以產生 Google Maps 搜尋與導航 URL。
- 建立清楚的元件與服務層架構，保留未來接 Google Places API 與 AI API 的空間。

### 目前已經超出 MVP 的功能

後續開發中，專案已加入：

- Node.js + Express 後端。
- Google Places API 搜尋真實店家。
- Google Geocoding API 搜尋地點與反查縣市。
- Google Maps JavaScript API 地圖圖釘選點。
- OpenAI API 用於自然語言搜尋詞轉換與餐廳排序。
- 收藏與歷史紀錄。
- 扭蛋隨機抽動畫。
- RWD 手機版直式、電腦版橫式排版。
- Vercel + Render 部署方向。

## 4. 使用者情境

### 情境一：不知道今天要吃什麼

- 使用者下課後肚子餓，但不知道要吃飯、麵、火鍋還是甜點。
- 使用者打開 PickEat，輸入「想吃熱湯」或「不要太油」。
- PickEat 依照需求推薦附近餐廳，讓使用者不用自己一直滑 Google Maps。

### 情境二：有價格區間限制

- 學生想控制餐費，希望不要吃太貴。
- 使用者選擇「可能百元內」或「平價，約 100～200 元」。
- 系統會依照價格區間篩選店家，讓推薦結果更符合學生預算。

### 情境三：想找附近餐廳

- 使用者希望找 1 公里內的餐廳，不想走太遠。
- 使用者選擇距離，或自訂距離。
- PickEat 會根據目前位置或地圖圖釘位置查詢附近店家。

### 情境四：想用特定地點當搜尋中心

- 使用者不一定想用目前定位，也可能想找學校、車站、某個景點附近的餐廳。
- 使用者可以輸入「彰師大」或選擇縣市後拖曳圖釘。
- 地圖圖釘位置會成為搜尋附近餐廳的根據地。

### 情境五：還是選不出來

- 使用者看到多間推薦後還是無法決定。
- 可以按「扭蛋隨機抽」。
- 系統會從符合條件的餐廳中抽出一間，讓選擇困難的人直接得到答案。

### 情境六：和朋友討論餐廳

- 使用者搜尋後看到幾間適合聚餐的餐廳。
- 可以點「查看地圖」分享位置，或按「開始導航」前往。
- 也可以把店家收藏，之後再回來查看。

## 5. 網站主要功能介紹

> 以下功能依目前程式碼實際存在的內容整理。這個專案目前是單頁式 React 應用，沒有獨立詳細頁，餐廳資訊集中顯示在餐廳卡片中。

### 5.1 首頁與主視覺

- **功能說明**：首頁顯示 PickEat 品牌名稱與副標題「今天吃什麼？讓 PickEat 幫你快速決定」。
- **使用方式**：使用者進入網站後，左側或上方會看到搜尋條件，右側或下方看到推薦結果區。
- **解決問題**：讓使用者一進網站就知道這是一個餐廳推薦工具。
- **程式實作**：主要在 `src/App.jsx` 中渲染，視覺樣式在 `src/App.css`。目前採用靛色、深藍、霓虹、像素與 vaporwave 風格，並有 glitch 標題、網格背景、掃描線與切角卡片。

### 5.2 搜尋根據地與地圖圖釘

- **功能說明**：使用者可以選擇縣市、輸入地點或直接拖曳地圖圖釘，設定要搜尋餐廳的中心點。
- **使用方式**：
  1. 選擇縣市。
  2. 或輸入「彰師大」、「台中車站」、「台北 101」等所在地。
  3. 在 Google 地圖上點擊或拖曳圖釘。
- **解決問題**：單純選縣市太粗略，無法真正代表附近餐廳。加入地圖圖釘後，搜尋會更接近使用者想找的區域。
- **程式實作**：
  - `src/components/GoogleMapPicker.jsx`：載入 Google Maps JavaScript API，建立地圖和可拖曳 Marker。
  - `src/components/SearchPanel.jsx`：顯示縣市選單、地點輸入、地圖選點。
  - `src/data/taiwanCities.js`：存放全台灣縣市中心座標。
  - `server/services/geocodingService.js`：後端呼叫 Google Geocoding API，支援地點搜尋與定位反查縣市。

### 5.3 GPS 定位

- **功能說明**：按下「定位我的位置」後，瀏覽器會要求定位權限，取得使用者目前座標。
- **使用方式**：使用者點擊「定位我的位置」，並允許瀏覽器取得位置。
- **解決問題**：使用者不用手動選地點，網站可以直接用目前位置找附近餐廳。
- **程式實作**：`src/App.jsx` 中的 `handleLocate()` 使用 `navigator.geolocation.getCurrentPosition()`，成功後呼叫後端 reverse geocode 取得縣市資訊；失敗時保留預設彰化位置與提示。

### 5.4 價格區間篩選

- **功能說明**：使用者可以依照價格區間篩選餐廳。
- **目前選項**：
  - 不限
  - 可能百元內
  - 平價，約 100～200 元
  - 中等，約 200～300 元
  - 中高，約 300～400 元
  - 偏高，約 400～500 元
  - 高價，500 元以上
  - 自訂價格上限
- **解決問題**：學生用餐通常會考慮預算，因此價格是很重要的推薦條件。
- **程式實作**：
  - `src/components/SearchPanel.jsx`：提供價格選單和自訂價格輸入。
  - `src/services/priceService.js`：前端判斷餐廳是否符合價格條件。
  - `server/services/priceService.js`：後端將 Google Places 的 `priceLevel` 轉成更細的中文價格區間。

### 5.5 距離篩選

- **功能說明**：使用者可以選擇 1 公里內、2 公里內、5 公里內、不限或自訂距離。
- **使用方式**：在距離選單中選擇範圍；若選自訂，可以輸入公里數。
- **解決問題**：餐廳再好，如果太遠也不一定適合當下需求。
- **程式實作**：
  - `src/App.jsx` 中的 `getRadiusMeters()` 將距離轉成 Google API 查詢半徑。
  - `applyRealRestaurantFilters()` 會再用餐廳距離做前端過濾，避免 1 公里內卻顯示 2 公里外的店家。
  - `server/services/googlePlacesService.js` 用座標計算餐廳與搜尋中心的距離。

### 5.6 需求輸入與快速標籤選單

- **功能說明**：使用者可以輸入自然語言需求，也可以從快速需求選單加入標籤。
- **目前標籤類型**：
  - 情境需求：熱湯、便宜、一個人、清爽、不油膩、吃很飽、快速、近學校。
  - 餐點類型：便當、雞肉飯、滷肉飯、牛肉麵、拉麵、火鍋、韓式、咖哩、早午餐、素食、飲料甜點。
  - 用餐場景：朋友聚餐、宵夜、早八早餐、下課晚餐、下午茶、健康餐。
- **解決問題**：有些使用者不知道怎麼描述需求，快速標籤可以降低輸入門檻。
- **程式實作**：
  - `src/components/SearchPanel.jsx` 的 `quickTagGroups` 管理標籤。
  - 標籤加入後會寫進 `filters.query`。
  - `src/services/recommendationService.js` 在示範資料模式下用關鍵字規則計分。
  - 後端 AI 模式會用 OpenAI 把自然語言轉成適合 Google Places Text Search 的搜尋詞。

### 5.7 Google Places 真實餐廳搜尋

- **功能說明**：目前前端會透過後端呼叫 Google Places API 搜尋真實店家資料。
- **使用方式**：使用者按下「幫我推薦」，前端把座標、距離、價格和需求送到後端。
- **解決問題**：正式版不應只靠假資料，Google Places 可以提供真實店名、地址、評分、營業狀態、評論與地圖連結。
- **程式實作**：
  - `src/services/placeApiService.js`：前端呼叫 `/api/recommendations`。
  - `server/index.js`：提供 `/api/recommendations`、`/api/restaurants/search`、`/api/places/nearby`。
  - `server/services/googlePlacesService.js`：呼叫 Places API，欄位包含店名、地址、評分、價格等級、營業狀態、評論等。

### 5.8 OpenAI AI 推薦

- **功能說明**：如果後端有設定 `OPENAI_API_KEY`，系統會使用 OpenAI 協助處理搜尋與推薦。
- **使用方式**：使用者不需要額外操作，只要正常輸入需求並按推薦。
- **解決問題**：使用者輸入的自然語言可能很口語，例如「今天很冷想喝熱湯，不要太貴」。AI 可以把這種描述轉成較適合搜尋與排序的條件。
- **程式實作**：
  - `server/services/aiRecommendationService.js`：
    - `createAiSearchQuery()`：把使用者需求轉成 Google Places 搜尋詞。
    - `rankRestaurantsWithAi()`：從 Google Places 回傳餐廳中挑出最多 4 間，並產生簡短推薦原因。
  - AI Key 放在後端 `.env`，不放在 React 前端，避免金鑰外洩。
  - 若沒有 OpenAI Key 或 AI 呼叫失敗，系統會使用 fallback 推薦原因。

### 5.9 推薦結果卡片

- **功能說明**：每間餐廳以卡片顯示。
- **目前顯示內容**：
  - 營業狀態
  - 收藏按鈕
  - 餐廳名稱
  - 價格區間
  - 距離
  - 評分
  - 地址
  - 兩則中文好評摘錄或推薦原因
  - 查看地圖
  - 開始導航
- **解決問題**：把使用者最需要判斷的資訊集中在同一張卡片中。
- **程式實作**：
  - `src/components/RestaurantCard.jsx`：處理卡片 UI。
  - 會過濾英文內容，避免 Google Places 類型標籤或英文評論影響中文介面。
  - 目前已移除餐廳 category 標籤、英文標籤、留言下方的額外標籤，讓卡片更乾淨。

### 5.10 評論好評摘錄

- **功能說明**：餐廳卡片下方會顯示最多 2 則短版中文好評。
- **使用方式**：使用者搜尋餐廳後，在卡片下方直接看到好評片段。
- **解決問題**：比起空泛的推薦句，真實評論片段更能讓使用者產生想去吃的感覺。
- **程式實作**：
  - `server/services/googlePlacesService.js` 會從 Google Places `reviews` 中整理評論，去除太短、英文或不適合顯示的內容。
  - 目前同一家店每次重新搜尋時，後端會從可用好評中隨機挑 2 則。
  - `src/components/RestaurantCard.jsx` 再做前端顯示與清理。

### 5.11 扭蛋隨機抽

- **功能說明**：如果使用者還是不知道要吃什麼，可以使用扭蛋機動畫隨機抽一間餐廳。
- **使用方式**：按下「扭蛋隨機抽」。
- **解決問題**：專門處理選擇困難，讓網站不只是理性篩選，也有趣味互動。
- **程式實作**：
  - `src/components/GachaMachine.jsx`：扭蛋機 UI。
  - `src/App.jsx` 的 `handleRandomRecommend()`：從符合條件的候選餐廳中抽一間，並優先從營業中的店家抽。
  - CSS 中有扭蛋機機身、膠囊、轉動動畫與色彩設計。

### 5.12 收藏功能

- **功能說明**：使用者可以按愛心收藏餐廳，收藏會保存在瀏覽器。
- **使用方式**：在餐廳卡片上按「收藏」或「已收藏」。
- **解決問題**：看到不錯的餐廳可以先存起來，不需要重新搜尋。
- **程式實作**：
  - `src/App.jsx` 用 `localStorage` 保存收藏 ID 和餐廳快照。
  - `src/components/FavoriteList.jsx` 顯示收藏清單。
  - 早期曾遇到「收藏只留下編號，沒有完整店家資料」的問題，後來改成保存餐廳快照，避免重新整理後只剩 ID。

### 5.13 推薦歷史紀錄

- **功能說明**：網站保存最近 5 次推薦或扭蛋抽選結果。
- **使用方式**：推薦或抽選後，下方「推薦歷史紀錄」會自動更新；可按清除紀錄。
- **解決問題**：使用者可以回頭看剛剛搜尋過哪些條件與餐廳。
- **程式實作**：
  - `src/App.jsx` 的 `addHistoryItem()` 建立歷史紀錄。
  - `src/components/HistoryList.jsx` 顯示最近 5 筆。
  - 使用 `localStorage` 保存，不需要資料庫。

### 5.14 查看地圖與開始導航

- **功能說明**：餐廳卡片提供「查看地圖」與「開始導航」。
- **使用方式**：點擊按鈕後，開啟 Google Maps。
- **解決問題**：推薦結果不只停在網站內，還能接到實際前往餐廳的流程。
- **程式實作**：
  - `src/services/mapService.js`：
    - `createSearchUrl(query)` 產生 Google Maps 搜尋連結。
    - `createDirectionsUrl(destination)` 產生 Google Maps 導航連結。
  - 若 Google Places 回傳 `googleMapsUri`，卡片會優先使用真實 Google 地圖網址。

### 5.15 備援示範資料

- **功能說明**：如果後端、Google Places 或 AI 暫時不可用，前端會改用 AI 生成的彰化市虛構餐廳資料。
- **使用方式**：使用者不需要手動切換，錯誤時自動 fallback。
- **解決問題**：展示時就算 API 失敗，網站仍然能展示核心概念，不會整個空白。
- **程式實作**：
  - `src/data/restaurants.js`：22 筆虛構彰化市風格餐廳。
  - `src/services/recommendationService.js`：示範資料推薦邏輯。
  - `src/App.jsx` 的 catch 區塊會呼叫 `getFallbackResults()`。

### 5.16 RWD 手機版與電腦版

- **功能說明**：電腦版採左右分欄；手機版改為直式排列。
- **使用方式**：使用者用手機或電腦打開同一網址，版面會自動調整。
- **解決問題**：餐廳推薦很常在手機上使用，因此手機版可讀性很重要。
- **程式實作**：
  - `src/App.css` 最後加入 responsive layout guard。
  - `1024px` 以上維持左側搜尋、右側結果。
  - `1023px` 以下改成上下排列，避免手機出現橫向版面。

## 6. 技術架構

### 6.1 前端技術

- **React 19**：負責建立元件化 UI。

- **Browser Geolocation API**：取得使用者 GPS。
- **Google Maps JavaScript API**：在前端顯示地圖並拖曳圖釘。

- **Node.js + Express**：建立後端 API。

目前沒有資料庫。

原因是這個專案先以展示核心功能為主，收藏和歷史紀錄先使用瀏覽器 `localStorage`，不需要登入和伺服器儲存。這樣可以降低開發難度，也方便課堂展示。

未來如果要支援多裝置同步、會員系統、收藏雲端保存，就可以加入資料庫。

### 6.4 是否使用 API

目前有使用或準備使用以下 API：

- Google Places API：搜尋真實餐廳。
- Google Geocoding API：搜尋地點與反查座標。
- Google Maps JavaScript API：前端地圖選點。
- Google Maps URL：查看地圖與開始導航。
- OpenAI API：解析需求、產生搜尋詞、排序餐廳與推薦原因。

### 6.5 API Key 保護方式

Google Places API Key 和 OpenAI API Key 不放在前端，而是放在 `server/.env`。前端只呼叫自己的後端 API，後端再呼叫 Google 或 OpenAI。

這樣做的原因是：

- 避免 API Key 被使用者在瀏覽器看到。
- 可以集中處理錯誤。
- 可以設定 CORS 限制來源。
- 未來可以加入流量控管、紀錄和安全規則。

## 7. 開發流程與修改紀錄

### 第一階段：確定專案主題

一開始確定主題是「今天吃什麼推薦網站」，名稱為 PickEat。核心想法是讓使用者選擇地點、價格區間、距離並輸入需求，網站推薦餐廳，幫助有選擇困難的人。

這個階段先確立 MVP 範圍：不要一開始做完整正式版，先做可以操作、可以展示核心概念的版本。

### 第二階段：建立基本網站架構

最初建立 React + Vite + JavaScript 專案，並依照初學者容易理解的方式拆分檔案：

- `data/restaurants.js`：餐廳假資料。
- `services/mapService.js`：Google Maps URL。
- `services/recommendationService.js`：推薦邏輯。
- `components/SearchPanel.jsx`：搜尋表單。
- `components/RestaurantCard.jsx`：餐廳卡片。
- `App.jsx`：整合狀態與畫面。

第一版使用 22 筆彰化市風格虛構餐廳資料，包含飯類、雞肉飯、滷肉飯、牛肉麵、拉麵、火鍋、韓式、咖哩、早午餐、素食、甜點、宵夜等情境。

這個階段的重點是：先讓網站能跑、能展示推薦流程，不急著串正式 API。

### 第三階段：加入核心推薦功能

接著加入價格、距離和需求文字推薦邏輯。

示範資料模式會：

- 依價格區間篩選。
- 依距離篩選。
- 根據「熱湯、便宜、一個人、清爽、不油膩、聚餐、宵夜」等文字計分。
- 優先排序營業中、符合關鍵字、評分高、距離近的餐廳。

後來因為使用者希望一次不要推薦太多，推薦結果改成最多顯示 4 間，讓畫面更聚焦，也更符合「快速決定」的目的。

### 第四階段：加入扭蛋隨機模式

使用者提出想要「隨機模式，要像扭蛋動畫」。因此新增 `GachaMachine.jsx`，用扭蛋機視覺讓使用者在選不出來時可以抽一間。

後來又調整成：

- 扭蛋優先從營業中的店家抽。
- 如果後端可用，會從 Google Places 真實店家中抽。
- 如果後端不可用，會退回示範資料。
- 扭蛋機外觀也從一般色彩改成更像扭蛋機的特色配色。

這個功能讓網站更有記憶點，也更符合「選擇困難」主題。

### 第五階段：加入收藏與歷史紀錄

後來加入收藏功能，卡片上有愛心圖標，使用者可以收藏喜歡的餐廳。

一開始收藏曾出現問題：重新整理後收藏只留下編號，缺少完整餐廳資料。後來改成同時保存：

- 收藏 ID：用來判斷是否已收藏。
- 餐廳快照：用來重新整理後仍能顯示完整卡片。

接著加入推薦歷史紀錄，保存最近 5 次推薦或扭蛋抽選結果。這讓使用者可以回顧剛剛查過什麼，也適合簡報 Demo。

### 第六階段：改善篩選標籤

原本快速標籤數量較少，後來逐步增加更多情境、餐點與用餐場景。再後來因為標籤太長、畫面太擠，改成下拉選單形式。

這個修改讓 UI 更乾淨，也避免手機版畫面被大量標籤撐開。

### 第七階段：建立後端並保護 API Key

當專案要從假資料往正式資料邁進時，開始建立 `server/` 後端。

後端使用：

- Express
- cors
- dotenv
- Google Places API
- Google Geocoding API
- OpenAI SDK

後端的目的不是做複雜資料庫，而是先解決兩件事：

1. 保護 Google / OpenAI API Key，不讓它出現在前端。
2. 把 Google 回傳資料整理成前端卡片可以使用的格式。

這個階段也補上 `.env.example` 和 `.gitignore`，避免 `.env`、`.env.local`、`server/.env`、`node_modules` 被上傳到 GitHub。

### 第八階段：串接 Google Places 與地點搜尋

後端建立後，前端改成優先呼叫後端 `/api/recommendations`。後端再呼叫 Google Places API 搜尋餐廳。

開發時遇到 Google API 欄位與請求格式問題，例如 Places API 不同 endpoint 對 `locationRestriction` 的格式要求不同。後來 Text Search 使用矩形範圍，Nearby Search 使用 circle，並在前端再做距離過濾，解決「設定 1 公里內卻出現 2 公里外餐廳」的問題。

後來也加入 Geocoding 功能，讓使用者可以輸入「彰師大」這種地點，把地圖圖釘移到該位置。縣市會依照定位或地點資料同步更新。

### 第九階段：加入 Google 地圖圖釘選點

使用者覺得單純選縣市沒有用，因為選縣市範圍太大。於是把原本的生活圈選擇改成：

- 縣市選擇。
- 搜尋所在地。
- Google 地圖。
- 可點擊或拖曳的圖釘。

這樣使用者可以自己決定「以哪個地方為中心搜尋附近餐廳」，比只有縣市更實用。

### 第十階段：整合 OpenAI AI 推薦

後來開始思考 AI 要如何整合。最後採用的方向是：AI 不取代 Google Places，而是協助理解需求和排序。

目前後端 AI 流程是：

1. 使用者輸入自然語言需求。
2. 後端用 OpenAI 將需求轉成適合 Google Places 的搜尋詞。
3. 後端用 Google Places 搜尋真實餐廳。
4. 後端再用 OpenAI 從餐廳清單中挑出最多 4 間並產生推薦原因。
5. 前端顯示餐廳卡片。

這樣做的好處是：真實店家資料仍由 Google 提供，AI 負責「理解人話」和「幫忙排序」。

### 第十一階段：改善餐廳卡片內容

卡片內容經過多次調整：

- 移除 `restaurant.category` 類型標籤。
- 把 Google Places 英文類型轉成中文。
- 後來再移除餐廳、餐飲等無意義標籤。
- 移除留言下方的標籤，讓好評區更乾淨。
- 把價格感改成「價格區間」。
- 評論改成擷取最多 2 則中文好評。
- 同一家店每次重新搜尋時，隨機挑 2 則可用好評。

這些修改都是為了讓結果卡片更像真正使用者會看的餐廳推薦，而不是單純展示資料欄位。

### 第十二階段：調整 UI / UX 與網站風格

網站風格經過多輪修改：

1. 初期是簡潔科技感、玻璃擬態卡片。
2. 後來嘗試 cyberpunk/glitch 設計系統。
3. 使用者不喜歡粉色，改成偏藍色。
4. 再試過 Midnight Ramen Blue、Ice Terminal。
5. 最後朝深藍、靛色、vaporwave、像素感、多層次霓虹方向調整。

目前視覺包含：

- 深色靛色背景。
- 網格與掃描線。
- 霓虹邊框。
- glitch 標題。
- 切角卡片。
- 多色但偏冷色調的層次。
- 手機版直式、電腦版橫式。

這個階段的重點是讓網站更有品牌感，也更適合簡報展示。

### 第十三階段：部署與 GitHub 協作

專案後來推上 GitHub，並討論如何讓同學下載、編輯和再上傳。

過程中處理了：

- 不上傳 `node_modules`。
- 不上傳 `.env`。
- GitHub push 認證問題。
- Apple 登入 GitHub、PAT token、repo 權限問題。
- commit author 顯示成 EDIS-tech 導致 Vercel 擋部署。
- 後來用 `git config user.name` 和 `git config user.email` 修正 commit 作者，並 amend + force push。

部署方向：

- 前端部署到 Vercel。
- 後端部署到 Render。
- Vercel 需要設定 `VITE_API_BASE_URL` 指向 Render 後端。
- Render 需要設定 `ALLOWED_ORIGINS` 允許 Vercel 網址。

### 第十四階段：手機版排版修正

使用者發現手機版打開網站時不是直式排版，因此最後加入響應式規則：

- `1024px` 以上：電腦版橫式，左側搜尋，右側結果。
- `1023px` 以下：手機和平板直式，上方搜尋，下方結果。

這個修改讓手機使用情境更合理，也避免小螢幕硬塞橫向版面。

目前網站已完成：

## 8. 開發過程中遇到的問題

### 問題一：一開始不知道功能要怎麼切

- **問題描述**：最初目標很多，例如 Google Places、定位、導航、AI、部署，但如果全部一次做會太複雜。
- **發生原因**：正式版功能很多，學生專題時間有限。
- **解決方式**：先定義 MVP，只做前端、假資料、篩選、推薦卡片與地圖 URL，再逐步擴充。
- **學到的東西**：開發時要先做最小可行版本，先讓核心流程跑起來，再慢慢增加功能。

### 問題二：專案路徑與啟動位置錯誤

- **問題描述**：一開始在 Terminal 輸入 `cd pickeat`、`npm install`，但出現找不到 `package.json`。
- **發生原因**：當下所在資料夾不是專案根目錄。
- **解決方式**：確認專案位置，後來將 `pickeat` 放到桌面並使用正確路徑 `/Users/user/Desktop/pickeat`。
- **學到的東西**：執行 npm 指令前，要先確認自己在有 `package.json` 的專案根目錄。

### 問題三：API Key 不能放在前端

- **問題描述**：如果把 Google 或 OpenAI API Key 放進 React 前端，其他人可能在瀏覽器看到。
- **發生原因**：前端打包後的程式會被下載到使用者瀏覽器。
- **解決方式**：建立 Express 後端，把 API Key 放在 `server/.env`，前端只呼叫自己的後端。
- **學到的東西**：涉及金鑰、付費服務或敏感資料時，需要後端保護。

### 問題四：Google Places API 欄位與請求格式

- **問題描述**：曾遇到 Google Places 回傳錯誤，例如 `locationRestriction` 的 `circle` 欄位不適用於某個請求。
- **發生原因**：Google Places API 不同方法需要不同 request body 格式。
- **解決方式**：Text Search 改用 rectangle 範圍，Nearby Search 保留 circle；前端再額外用距離條件過濾。
- **學到的東西**：串 API 不能只看概念，要依照官方格式調整，而且需要錯誤處理。

### 問題五：距離篩選不準確

- **問題描述**：使用者設定 1 公里內，但結果可能出現 2 公里外的餐廳。
- **發生原因**：Google Places 回傳結果不一定完全符合前端想像的範圍。
- **解決方式**：前端拿到 Google 結果後再用 `restaurant.distance <= distanceLimit` 做二次過濾。
- **學到的東西**：即使 API 有查詢條件，前端仍需要做自己的資料驗證與過濾。

### 問題六：單純選縣市不夠實用

- **問題描述**：只選擇縣市對餐廳搜尋幫助有限，因為縣市範圍太大。
- **發生原因**：使用者真正需要的是以某個具體位置為中心找附近餐廳。
- **解決方式**：加入 Google 地圖圖釘、地點搜尋與 GPS 定位。
- **學到的東西**：功能要根據真實使用情境調整，而不是只做表面選單。

### 問題七：收藏重新整理後資料不完整

- **問題描述**：收藏列表顯示有幾間，但看不到完整店家卡片。
- **發生原因**：只保存 ID，重新整理後可能找不到對應完整資料。
- **解決方式**：除了 ID，也保存餐廳快照到 `localStorage`。
- **學到的東西**：本機資料保存不能只存索引，還要考慮重新整理後是否能還原畫面。

### 問題八：UI 標籤太多、畫面太擠

- **問題描述**：快速篩選標籤越加越多，畫面變長、不好看。
- **發生原因**：直接把所有標籤顯示在畫面上會佔太多空間。
- **解決方式**：改成分組下拉選單，只顯示已選標籤。
- **學到的東西**：功能變多時，UI 要重新設計，不只是一直往畫面上加東西。

### 問題九：推薦原因太空泛

- **問題描述**：原本推薦原因看起來不夠吸引人，也不像真的餐廳介紹。
- **發生原因**：單靠固定模板容易寫出重複、空泛的句子。
- **解決方式**：改成抓取 Google Places 評論中的中文好評短句，並限制長度，避免長篇複製。
- **學到的東西**：推薦不只要有資料，還要讓使用者覺得有參考價值。

### 問題十：英文類型標籤影響中文介面

- **問題描述**：Google Places 會回傳 `japanese_restaurant`、`food`、`establishment` 等英文類型。
- **發生原因**：Google API 原始資料不一定適合直接顯示給使用者。
- **解決方式**：建立翻譯與隱藏規則，後來進一步移除無意義標籤。
- **學到的東西**：API 回傳資料需要經過整理，不能全部原封不動顯示。

### 問題十一：Vercel 部署權限問題

- **問題描述**：Vercel 顯示 commit author 沒有專案貢獻權限，Hobby Plan 不支援 private repo 協作。
- **發生原因**：git commit 作者曾顯示為 EDIS-tech，不是 Vercel 專案認可的帳號。
- **解決方式**：設定本機 git user name/email，重新 amend commit 並 force push。
- **學到的東西**：部署平台會檢查 GitHub 權限與 commit author，協作時要注意帳號設定。

### 問題十二：線上前端看起來沒有連到後端

- **問題描述**：網站線上版可能出現後端沒有啟用的感覺。
- **發生原因**：Vercel 前端需要設定 `VITE_API_BASE_URL` 指到 Render 後端；Render 也要設定 `ALLOWED_ORIGINS`。
- **解決方式**：在 Vercel 設定 `VITE_API_BASE_URL=https://pickeat-l4ud.onrender.com`，Render 設定允許 Vercel 網址。
- **學到的東西**：前後端分開部署時，除了程式碼，環境變數和 CORS 也很重要。

### 問題十三：手機版排版不是直式

- **問題描述**：手機開網站時仍看起來像橫式排版，不符合手機使用習慣。
- **發生原因**：CSS 多組響應式規則互相覆蓋，部分 grid 還保留桌機邏輯。
- **解決方式**：新增最後一層 responsive layout guard，強制 1023px 以下全部直式。
- **學到的東西**：RWD 不只是縮小字體，還要重新安排資訊順序。

## 9. 解決方法與設計決策

### 9.1 為什麼先做前端 MVP

- **原因**：一開始如果直接串 Google、AI、後端和部署，開發成本太高，也容易卡在 API 設定。
- **好處**：先完成可操作版本，能快速驗證專題概念，也方便後續逐步擴充。

### 9.2 為什麼先用假資料

- **原因**：假資料可以完全控制欄位，例如價格、距離、評分、關鍵字和推薦原因。
- **好處**：不用一開始處理 API Key、帳單、格式錯誤，就能先展示網站核心流程。

### 9.3 為什麼不用資料庫

- **原因**：目前功能以展示為主，收藏和歷史紀錄可以先存在使用者瀏覽器。
- **好處**：降低架構複雜度，避免一開始就處理登入、資料表和伺服器儲存。

### 9.4 為什麼推薦結果用卡片呈現

- **原因**：餐廳資訊包含店名、價格、距離、評分、地址、評論和按鈕，卡片最適合分組。
- **好處**：使用者可以快速比較多間餐廳，也方便手機版一張一張閱讀。

### 9.5 為什麼推薦最多 4 間

- **原因**：如果一次顯示太多餐廳，使用者還是會選擇困難。
- **好處**：最多 4 間讓結果更聚焦，也符合「幫我快速決定」的產品定位。

### 9.6 為什麼要有扭蛋模式

- **原因**：有些人即使看到推薦還是無法決定。
- **好處**：扭蛋模式增加趣味性，也讓網站更符合選擇困難的主題。

### 9.7 為什麼要建立後端

- **原因**：Google Places 和 OpenAI API Key 不能放在前端。
- **好處**：後端可以保護金鑰、整理資料、處理錯誤，也讓未來擴充更安全。

### 9.8 為什麼 AI 不取代 Google Places

- **原因**：AI 可能會產生不存在的店家，不適合當真實餐廳資料來源。
- **好處**：Google Places 負責真實資料，AI 負責理解需求和排序，角色分工更合理。

### 9.9 為什麼要使用地圖圖釘

- **原因**：縣市選擇太粗略，無法代表使用者真正想搜尋的位置。
- **好處**：圖釘讓使用者可以直接指定搜尋中心，例如學校、車站、商圈。

### 9.10 為什麼要做手機版直式

- **原因**：餐廳搜尋通常發生在手機上。
- **好處**：手機版直式可以降低橫向滑動和閱讀困難，使用體驗更自然。

## 10. 專案亮點

- 解決每天都會遇到的真實生活問題：不知道今天吃什麼。
- 使用者可以依地點、距離、價格區間、需求文字推薦餐廳。
- 支援 GPS 定位、縣市選擇、地點搜尋和地圖圖釘。
- 前端有完整互動流程，不是靜態頁面。
- 後端保護 API Key，架構比純前端更接近正式產品。
- Google Places 提供真實店家資料，示範資料作為 fallback。
- OpenAI 協助自然語言需求解析與餐廳排序。
- 推薦結果最多 4 間，避免資訊過載。
- 扭蛋隨機抽模式有趣，符合選擇困難情境。
- 收藏與歷史紀錄讓網站更像真正可用的工具。
- 視覺風格有靛色科技感、像素、霓虹與 glitch，不像普通表單網站。
- 手機版和電腦版有不同排版，適合實際展示。
- 從 MVP 到正式版功能逐步擴充，開發歷程完整，適合課堂報告。

## 13. 個人心得與學到的東西

透過這次 PickEat 專題，我學到一個網站不只是把畫面做出來而已，而是要先從生活中真的會遇到的問題開始思考。原本只是想解決「今天吃什麼」這個很日常的困擾，但在開發過程中才發現，使用者其實會考慮很多條件，例如地點、距離、價格、營業狀態、心情和用餐情境。

我也學到 MVP 的概念很重要。一開始不用急著把所有正式功能都做完，而是先做出可以操作的版本，再依照使用過程中發現的問題慢慢修改。像是先用假資料完成推薦流程，後來再加入後端、Google Places、AI、收藏、歷史和地圖圖釘，這樣比較不會一開始就卡住。

在技術上，我學到 React 元件拆分、前端狀態管理、localStorage、API 串接、後端 Express、環境變數、API Key 保護和部署設定。也學到 UI/UX 需要一直調整，因為有些功能做出來雖然可以用，但實際看起來可能太擠、不直覺，或手機版不適合使用。

這個專案讓我體會到開發網站要先求能跑，再求好用，最後再慢慢把畫面和功能優化。報告時也不能只展示畫面，還要說清楚為什麼做這個功能、過程中遇到什麼問題、後來怎麼解決。

## 口頭報告講稿草稿

大家好，我們這次的專題是 PickEat，主題是「今天吃什麼推薦網站」。我們會想做這個題目，是因為生活中常常遇到不知道要吃什麼的情況，尤其是學生下課後，常常會想找便宜、附近、適合當下心情的餐廳，但自己打開地圖慢慢查其實很花時間。

PickEat 的核心概念是讓使用者輸入條件，例如搜尋地點、距離、價格區間，還有今天想吃什麼，網站就幫使用者推薦適合的餐廳。使用者可以用 GPS 定位，也可以選擇縣市或在地圖上拖曳圖釘，指定想搜尋的中心點。

一開始我們先做 MVP，也就是最小可行產品。第一版沒有串正式 API，而是先用彰化市風格的虛構餐廳資料，完成搜尋、篩選和推薦卡片。這樣可以先確認網站流程是可行的，不會一開始就卡在 API 或後端設定。

後來我們逐步加入更多功能，例如扭蛋隨機抽、收藏餐廳、推薦歷史紀錄。扭蛋功能是為了讓選擇困難的人可以直接抽一間，收藏和歷史則讓使用者可以保留自己看過或喜歡的餐廳。

接著我們建立了 Node.js 和 Express 後端，因為 Google Places 和 OpenAI 的 API Key 不能放在前端。前端會把使用者條件送到後端，後端再呼叫 Google Places 搜尋真實餐廳。如果有 OpenAI API Key，後端也會用 AI 把使用者輸入的自然語言轉成搜尋詞，並從餐廳清單中挑出最多 4 間推薦。

在開發過程中，我們也遇到一些問題。例如一開始單純選縣市不夠精準，所以後來加入地圖圖釘；距離篩選有時不準，所以前端又做一次距離過濾；收藏一開始只存 ID，重新整理後資料不完整，所以後來改成保存餐廳快照；手機版一開始不是直式排版，所以最後也針對 RWD 做了調整。

目前網站已經可以展示完整流程：使用者設定位置和條件，按下推薦，看到餐廳卡片，也可以換一批、扭蛋抽選、收藏、查看歷史，最後點 Google Maps 查看地圖或開始導航。

我覺得這個專案最大的收穫是，我們不是只做一個漂亮畫面，而是從真實生活問題開始，慢慢把功能拆開，一步一步做成可以使用的網站。未來如果再加入登入、資料庫和更完整的個人化推薦，PickEat 就可以更接近一個真正的餐廳推薦產品。
